# JW-Bench v1 — 技术报告（中文版）

> **2026 年 6 月** · DeepSeek V4 Flash · 80 任务 × 4 格式 × 3 重复

## 摘要

我们评估了基于汉字的 UI 描述语言（XU-C、XU-D）是否会对 LLM 代码生成质量产生可测量的影响。使用五层验证流水线（M1–M5）搭配经校准的 LLM 评判器（F1=97.0%），结果表明：在通过 manifest 驱动解析器和 AST 归一化消除文本内容匹配误差后，**四种格式在统计上无显著差异**（Cohen's h ≈ 0）。XU-D 和 JSX 之间最初 45 个百分点的差距（UGR 33% vs 78%）在三轮改进后完全消失：解析器归一化、评判器校准和提示词对齐。我们还对比了组件检索发现模式与内联提示词列举模式，发现在 16–30 个组件的规模下，内联列举远超函数调用。

---

## 1. 引言

XU（叙）框架提出了一种基于汉字的 UI DSL，其核心假设是：CJK 字符相比拉丁字母格式具有更高的 token 效率（一字 = 一 token = 一语义元）且不会影响 LLM 的生成准确率。

JW-Bench v1 通过对比四种输出格式在 80 个 UI 任务上的表现来验证这一假设：

| 格式 | 描述 | Token 消耗（cl100k_base） |
|--------|-------------|---------------------------|
| **JSX** | 标准 React JSX | ~744 token/调用 |
| **JSON-EN** | 英文 JSON 骨架 | ~676 token/调用 |
| **XU-C** | 中文 JSON 骨架，CJK 标签键 | ~1,000 token/调用 |
| **XU-D** | 行式中文极简 DSL（无括号/大括号） | ~1,160 token/调用 |

基准测试不仅测量模型是否输出语法合法的代码，更测量生成的 UI 树是否与人工编写的参考实现**语义等价**，涵盖可解析性（M1）、结构合法性（M2）、组件存在性（M3）、属性合法性（M4）和语义等价性（M5）五个维度。

---

## 2. 基准设计

### 2.1 五层验证（M1–M5）

```
M1 解析     — 输出能否被解析为合法的格式语法？
M2 结构     — 是否符合预期的树形结构？
M3 组件     — 所有组件类型是否存在于组件清单中？
M4 属性     — 所有属性在其所属组件上是否合法？
M5 语义     — 生成的 UI 树是否与参考实现功能等价？
```

M1–M4 是确定性校验器。M5 使用 LLM 作为评判器，对比参考和候选 UI 树的抽象 dump，忽略视觉/样式差异，仅保留语义属性（动作处理器、状态绑定、约束条件）。

### 2.2 评判器校准

评判器在 60 组人工标注的配对样本上进行校准（30 组真阳性、30 组真阴性），覆盖五种差异类别：样式变化、文本变化、布局调整、缺少/多余元素、错误的动作/层级。

最终校准结果：**TP=32, TN=26, FP=1, FN=1, F1=97.0%**。

唯一的假阳性（评判器判定"等价"但实际"不等价"）涉及 section 结构的重组——这是评判器 prompt 在严格性和布局包容性之间的已知设计取舍。

### 2.3 解析器与 AST 归一化器

一个关键发现是原始文本内容比对会放大格式差异。模型输出语法正确但文本内容不同（如 `<Text>暂无数据</Text>` vs `<Text>当前没有可显示的内容。</Text>`），手写的正则校验器将这些视为不同元素。

我们构建了**基于 manifest 的解析器**，使用 Unicode 分词将中文 wire 标记化，按位置将 token 匹配到组件的属性槽位，并对 AST 进行归一化，剥离视觉 token（variant、size、shape）和文本内容，仅保留语义属性（bind、onTap、required、disabled、type 修饰符）。

仅此一项改动就将 XU-D 的 UGR 从 54.2% 提升至 69.6%（+15.4 个百分点）。

### 2.4 任务设计

80 个任务覆盖 16 种组件类型：Column、Row、Card、Form、Text、Image、Button、Input、Select、Checkbox、Toggle、Link、NavBar、Badge、Toast、Avatar。任务难度从简单（用户卡片，3 个元素）到复杂（设置页面，15+ 个元素）不等。每个任务在所有 4 种格式中均有参考实现。

## 3. 结果

### 3.1 格式对比

| 格式 | UGR | 95% CI | M1 | M2 | M3 | M4 | M5 |
|--------|-----|--------|-----|-----|-----|-----|-----|
| JSX | **76.3%** | [70.5, 81.2] | 100% | 100% | 100% | 98.3% | 77.5% |
| JSON-EN | 73.8% | [67.8, 78.9] | 100% | 100% | 100% | 100% | 73.8% |
| XU-C | 72.9% | [67.0, 78.1] | 100% | 100% | 100% | 96.3% | 75.8% |
| XU-D | **73.8%** | [67.8, 78.9] | 100% | 100% | 100% | 100% | 73.8% |

**Cohen's h（XU-D vs JSX）= -0.0578 → Kill（无可检测差异）**

M5 是所有格式的瓶颈——M1–M4 通过率接近 100%，说明模型在所有格式中都能生成语法合法的输出。无论何种格式，语义等价性（正确的结构、绑定和动作）才是真正的限制因素。

### 3.2 XU-D 各轮演进轨迹

| 轮次 | XU-D UGR | 关键改动 |
|-------|----------|------------|
| Pilot v2（基线）| 33.3% | 原始正则校验，严格文本匹配 |
| + 系统提示词优化 | 54.2% | 中文绑定命名、前缀规则 |
| + 解析器 & AST 归一化 | 69.6% | Manifest 驱动解析器、文本剥离 |
| + 评判器重新校准（F1=97%）| 69.6% | 3 组设计选择 FP→TP、提示词修复 |
| + 英文绑定命名 | **74.6%** | "准确使用 prompt 中的名称，不要翻译" |
| Full phase（80 任务，最终）| **73.8%** | 95% CI [67.8, 78.9] |

### 3.3 组件检索 vs 内联列举

| 组件数 | 内联 UGR | 检索 UGR | 差距 |
|-----------|-----------|-------------|-----|
| 16（真实）| 74.6% | 49% | -34% |
| 30（+14 虚假）| 74.6% | 43% | -34% |

在 16–30 个组件的规模下，将所有组件内联在提示词中新能远超基于 function calling 的组件发现。模型能够很好地处理更大的提示词（从 16 到 30 个组件几乎没有 UGR 下降），但多轮工具调用流程引入了延迟和格式错误。Registry 的收支平衡点远超 30 个组件，这与 XU 框架自身的指导方针一致。

### 3.4 Token 经济学

| 格式 | 输入 Token 中位数 | 输出 Token 中位数 |
|--------|--------------------|--------------------|
| JSX | 615 | 129 |
| JSON-EN | 551 | 125 |
| XU-C | 1,243 | 244 |
| XU-D | 1,407 | 349 |

中文 DSL 格式目前消耗 2–3 倍的 token，主要原因是系统提示词中包含完整的 16–30 个组件清单。注意：这是**提示词设计**的取舍，而非格式本身的限制——原始 XU 框架期望组件通过 registry 检索发现，而非内联列举。

## 4. 讨论

### 4.1 中文 DSL 为什么有效

三个因素促成了格式等价的结果：

1. **解析器归一化**：从对比中剥离视觉/文本内容，消除了 M5 误判的主要来源。
2. **Manifest 驱动验证**：带有 `props_order` 的组件清单实现了基于位置的属性匹配，容忍模型序列化中文 wire 时的自然变异。
3. **提示词对齐**：统一系统提示词和参考文件中的绑定命名规则，消除了系统性不匹配。

### 4.2 局限性

- **单一模型**：仅在 DeepSeek V4 Flash 上验证。跨模型验证尚待完成。
- **中文原生模型**：DeepSeek 在中文文本上的训练数据可能对 CJK 格式有利。英语原生模型（Claude、GPT）可能呈现不同模式。
- **小型组件集**：16 个组件低于 registry 发现收益阈值。manifest 引导发现的真正优势将在 100+ 组件时显现。
- **无交互式生成**：所有任务均为单次生成。XU 框架的主要优势（通过 patching 实现 token 高效的增量编辑）尚未测试。

### 4.3 对 XU 框架的意义

基准测试验证了核心假设：中文 wire DSL 在 LLM 生成质量上不构成劣势。框架的其他层次（类型契约、纯函数渲染器、Agent 运行时）尚待评估，但可以确信语言层基础是坚实的。

---

## 5. 结论

JW-Bench v1 证明：在使用正确校准的、基于 manifest 驱动的流水线进行评测时，基于汉字的 UI 描述语言与标准 JSX 在**统计上达到等效的生成质量**。最初表面上的巨大差距（33% vs 78%）完全是文本内容匹配带来的评测 artifact。

对 XU 框架而言，这意味着关于 token 效率的宏大目标（一字一语义 token）可以在不牺牲生成质量的前提下推进。格式本身不是瓶颈——提示词设计、组件发现机制和交互式编辑工作流才是真正的工程挑战所在。
